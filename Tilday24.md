# Java 웹 프로그래밍: 서버 인프라와 자동화 빌드 (2026-01-09)

## 1. 서버 운영의 정석: WAR와 배포 메커니즘
선생님께서 코딩 기술보다 더 강조하신 것이 바로 **'배포(Deployment)'**의 안정성과 구조적 이해다.

* **WAR(Web Application Archive) 파일의 본질**: 
    - 우리가 개발한 자바 클래스, JSP, 설정 파일을 하나로 묶은 표준 압축 팩이다.
    - 톰캣(Tomcat)은 이 파일을 감지하면 자동으로 압축을 풀고 메모리에 올리는데, 이 과정을 **'자동 배포'**라고 부른다. 서버가 구동될 때 파일이 어떻게 로드되는지 이해하는 것이 백엔드의 시작이다.

* **IDE 중심의 운영**: 
    - 과거 방식처럼 서버 설정 파일(`server.xml` 등)을 직접 수정하는 것은 실수를 유발하는 위험한 방식이다. 
    - 대신 **IntelliJ**나 **Eclipse** 같은 IDE가 서버를 제어하게 함으로써, 개발자는 로직에 집중하고 설정 오류로 인한 사고를 방지하는 것이 현대적인 개발 트렌드다.

## 2. 웹 프로젝트의 설계도: web.xml (배포 서술자)
서버가 켜질 때 가장 먼저 읽어보는 **'프로젝트 가이드북'**이자 전체적인 경로를 제어하는 핵심 파일이다.

* **논리적 경로와 물리적 경로의 연결 (Mapping)**: 
    - 사용자가 브라우저에 입력하는 URL(논리적 경로)과 실제 서버에 존재하는 자바 파일(물리적 경로)은 별개다. 
    - 이 둘을 정확하게 이어주는 다리가 바로 `web.xml`의 **Servlet Mapping**이다. 이 연결이 어긋나면 서버는 요청을 처리할 대상을 찾지 못한다.

* **Context 설정의 중요성**: 
    - 웹 애플리케이션의 시작점(Root)을 정의한다. 이 설정이 꼬이면 이미지나 정적 자원의 경로가 모두 깨지는 대참사가 발생할 수 있다.

* **기술의 진화 (@WebServlet)**: 
    - XML 설정이 비대해지면 유지보수가 어려워지기 때문에, 자바 클래스에 직접 **어노테이션**을 붙여 직관적으로 경로를 관리하는 방식으로 발전하고 있다.

## 3. 라이브러리 관리의 혁신: Maven(메이븐)
라이브러리를 수동으로 관리하는 것은 비효율적이며, 메이븐을 통한 자동화가 필수적임을 강조하셨다.

* **메이븐 중앙 저장소**: 전 세계 자바 라이브러리가 체계적으로 관리되는 거대한 디지털 도서관이다.

* **의존성(Dependency) 관리**: 
    - `pom.xml`에 필요한 라이브러리 명세만 적으면 메이븐이 다운로드부터 빌드 경로 추가까지 자동으로 수행한다.
    - 특히 **'의존성의 전이(Transitive Dependency)'** 기능 덕분에, 특정 라이브러리가 요구하는 하위 라이브러리들까지 메이븐이 한꺼번에 관리하여 버전 충돌 문제를 최소화한다.

## 4. pom.xml 최적화와 표준 구조의 이해
프로젝트의 빌드 정보와 환경 설정을 담은 가장 중요한 명세서다.

* **JDK 버전 동기화 (필수 체크)**: 
    - 개발 환경의 JDK 버전과 `pom.xml`에 명시된 버전이 다르면 컴파일 에러가 발생한다. 
    - `<properties>` 태그 내의 소스 및 타겟 버전을 실제 구동 환경과 일치시키는 작업이 선행되어야 한다.

* **Servlet API 등록의 당위성**: 
    - 표준 자바(SE)에는 웹 관련 기능이 포함되어 있지 않다. 따라서 `javax.servlet-api`를 의존성에 추가해야만 `HttpServlet` 같은 클래스를 상속받아 웹 기능을 구현할 수 있다.

* **표준 디렉토리 구조 준수**: 
    - `src/main/java`: 백엔드 비즈니스 로직(Java)의 저장소.
    - `src/main/webapp`: 프론트엔드 자원(HTML, JSP, 정적 파일)의 저장소. 
    - 이 표준 구조를 지켜야 빌드 도구와 서버가 자원을 정상적으로 인식한다.
